# RLM (Recursive Language Models) アーキテクチャ移行タスク

## 背景と現状の課題
現在の `analyzer.py` は「Recursive Language Models (RLM)」を標榜していますが、実際には **「静的な深さ優先探索による階層的要約スクリプト」** に留まっています。

本来の RLM は MIT OASYS lab らが提唱する **「超長文コンテキストをプロンプトに入れるのではなく、REPL（外部環境）に変数の状態として保持し、LLM自身がコードを書いて探索・再帰呼び出しを行う推論パラダイム」** です。

現状のスクリプトは以下の点でRLMの要件を満たしていません。
1. **REPL (Sandbox) がない**: LLMがコードを実行して動的にプロジェクトを探索する環境が存在しない。
2. **Controller ループがない**: LLMが「Plan/Code生成 -> 実行 (Execute) -> 結果の観察 (Observe) -> 再帰 (Recurse)」を行う自律的な推論ループがない。
3. **`llm_query` の欠如**: LLM自身がサブモデルを呼び出し、その結果を文字列ではなくREPL内の「変数（状態）」として受け取る仕組みがない。

これらを解決し、真の RLM アーキテクチャを実現するためのタスクを以下に定義します。

---

## タスクリスト

### Phase 1: コアアーキテクチャの刷新
- [ ] **Task 1-1: Controller (推論ループ) の実装**
  - 現在の固定的なディレクトリ再帰探索 (`RLMAnalyzer.analyze`) を廃止し、LLMが最終目的（プロジェクト全体の解析）を達成するまで「コード生成 -> 実行 -> 観察」を繰り返す Agent ループ（Controller）を実装する。
- [ ] **Task 1-2: REPL (Sandbox) 環境の構築**
  - LLMが生成したPythonコードを安全に実行し、その状態（変数）を保持できる REPL（例: `LocalREPL` や専用のサンドボックス環境）を実装する。
  - プロジェクトのディレクトリ構造やファイル情報を、プロンプトのテキストとしてではなく、REPL内の初期オブジェクトや関数としてLLMに提供する。

### Phase 2: RLM特有の機能の実装
- [ ] **Task 2-1: 再帰呼び出しAPI (`llm_query`) の実装**
  - LLMがREPL環境内から「別のLLM（サブモデル）」を呼び出せる関数 `llm_query(prompt, context)` を実装する。
  - サブモデルの応答が親モデルのプロンプトを汚染しないよう、結果をREPL内の変数として返す仕組みを作る。
- [ ] **Task 2-2: プロンプトエンジニアリングの改修**
  - LLMに対して「あなたはPythonコードを記述してファイルシステムを探索し、必要に応じて `llm_query` を使って部分的な要約を行ってください」と指示するシステムプロンプトを作成する。

### Phase 3: 既存資産の再利用と統合
- [ ] **Task 3-1: Tree-sitter ロジックのツール化**
  - 現在の `analyzer.py` にある `tree-sitter` を用いたシンボル抽出ロジック（`_analyze_code_file`）を、LLMがREPL内から呼び出せるユーティリティ関数（例: `extract_symbols(filepath)`）として切り出して提供する。
- [ ] **Task 3-2: 構造化ドキュメンテーションロジックの適応**
  - 現在の Markdown ファイル出力の仕組みを、LLMの自律的な解析結果やREPL内の最終状態から `analysis_docs` を構築する形に統合・修正する。

### Phase 4: 検証と調整
- [ ] **Task 4-1: 無限ループ防止機構とコスト管理の導入**
  - LLMのコード実行エラーや再帰呼び出しの深さ、実行回数の上限（Max Steps / Max Depth）を設定し、無限ループを防ぐ仕組みを導入する。
  - 親モデルとサブモデルのAPI呼び出しコスト（トークン数）を統合して追跡するロジックを追加する。
- [ ] **Task 4-2: テストケースの作成 (`test_analyzer.py` の修正)**
  - 新しい RLM アーキテクチャに合わせた単体テスト・結合テストを作成し、既存のテスト（もしあれば）を移行する。
